# 决策顾问执行系统（30/60/90）

目标：在不牺牲可玩性的前提下，建立稳定推进节奏，并安全接入外部工具与 GitHub 开源能力。

适用：单人开发，每周 25h+。

---

## 执行原则（硬约束）

1. 先推进系统，再加工具；先验证收益，再并入主链路。
2. 每周最多引入 1 个外部能力（工具或开源项目），避免并发集成导致排障困难。
3. 所有外部接入必须可回退，并通过现有门禁脚本。
4. 工作时间遵循 `70/20/10`：
   - 70%：核心玩法/平衡/手感
   - 20%：推进系统与流程
   - 10%：外部工具/开源实验

---

## 0-30 天：推进系统落地（最高优先级）

### 目标
- 周节奏稳定运行。
- 所有任务可验收、可复盘。
- 外部接入有统一准入标准。

### 必做事项
- 建立周看板（可用 GitHub Projects 或本地看板）。
- 每周固定 5-7 个“可验收任务”，必须写出完成标准。
- 每次改动后执行：
  - `python verify_project.py`
  - 涉及战斗/关卡改动时执行 `python verify_project.py --full`
- 每周至少 1 次试玩复盘，记录核心指标：
  - 10 分钟存活率
  - 首次死亡时间
  - 升级选择犹豫点（出现在哪类选择）

### 交付件
- `docs\WEEKLY_25H_PLAYBOOK.md`
- `docs\EXTERNAL_INTEGRATION_POLICY.md`
- `docs\GITHUB_PROJECT_EVALUATION_CHECKLIST.md`

---

## 31-60 天：外部工具链接入（可观提速）

### 目标
- 构建“可观测 + 可验证”闭环。
- 减少线上/试玩阶段定位问题成本。

### 建议顺序
1. 错误监控：Sentry（先最小事件集）
2. 数据复盘：核心事件埋点 + 周报导出
3. 自动验证：GitHub Actions 最小门禁（验证脚本、基础构建）
4. 资产台账：第三方资源来源与许可证追踪

### 完成标准
- 任一错误可通过事件链路定位到具体模块。
- 每周有一次基于数据的数值调整，不靠纯主观。

---

## 61-90 天：GitHub 热门项目安全复用（增量收益）

### 目标
- 借力成熟能力，减少重复造轮子。
- 严控依赖污染和技术债。

### 优先方向（按收益排序）
1. 数据驱动/配置模板
2. AI 行为系统（状态机/行为树）
3. UI 动效反馈组件
4. 性能分析/调试工具

### 7 天接入评估节奏（每次仅一个候选）
1. Day1-2：隔离分支接入 Demo
2. Day3-4：主链路小范围试挂载
3. Day5：性能与稳定性对比
4. Day6：回退演练（确认可移除）
5. Day7：决策（保留/放弃）

### 一票否决
- 许可证不清晰
- 近 3-6 个月无活跃维护
- 无法在 7 天内证明净收益

---

## 周执行复盘模板（建议每周五）

- 本周完成了什么（只写可验证结果）
- 哪个任务最耗时，原因是什么
- 哪个外部接入真正带来效率提升
- 下周要删掉什么低收益事项
- 下周 5-7 个任务与验收标准
